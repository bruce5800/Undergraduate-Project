\section{System Design}
\label{sec:system}

Our system consists of two co-designed components:
(\S\ref{sec:simulator}) the AIGC inference simulator---an
open-source platform that models the five AIGC physics
enumerated in \S\ref{sec:intro} with realistic per-request
granularity---and (\S\ref{sec:rl}) the AIGC-aware RL
scheduler---a PPO policy that exposes AIGC physics to the agent
via structured state features and reward shaping.

\subsection{AIGC Inference Simulator}
\label{sec:simulator}

We implement the simulator in Python as a discrete-time
event-driven environment with a 0.1~s simulation step. Each
step performs four operations: (a) check for completed tasks
and release resources; (b) update task dependency and arrival
gating to transition \texttt{WAITING}~$\to$~\texttt{READY};
(c) invoke the scheduler to assign \texttt{READY} tasks to
servers; (d) start any admitted tasks on their target servers
and accumulate per-server energy. We organize the simulator's
physics modeling as four incremental milestones (M1--M4), each
addressing one or more of the five AIGC physics from
\S\ref{sec:intro}.

\textbf{M1: Model Weight Residency.} Every server maintains a
\texttt{loaded\_models} dictionary tracking which models have
weights in GPU VRAM, plus a \texttt{model\_refs} reference
count for currently running tasks that prevent eviction. When
a request arrives needing a model not yet loaded, the server
(i)~evicts the least-recently-used unpinned model(s) until VRAM
headroom suffices, and (ii)~pays a \texttt{cold\_load\_sec}
delay before the task starts execution. Cold-load times are
calibrated per model (e.g., 5~s for LLaMA-7B at 14~GB; 25~s
for LLaMA-70B INT8 at 70~GB) based on NVMe sequential read
bandwidth.

\textbf{M2: Prefill--Decode Two-Phase Tasks with KV Cache.}
Each inference request decomposes into a (Prefill, Decode)
pair sharing a \texttt{req\_id}. The Prefill task carries
\texttt{prompt\_tokens}; the Decode task carries
\texttt{output\_tokens} and depends on Prefill. We compute
task workload as $\text{prompt\_tokens} \cdot
\text{prefill\_tflops}/k$ and $\text{output\_tokens} \cdot
\text{decode\_tflops}/k$ per the published model architecture
parameters. The key design choice is \textbf{encoding KV cache
size into \texttt{prefill.output\_size}} (computed as
$\text{prompt\_tokens} \cdot \text{kv\_cache\_MB\_per\_token}
/1024$~GB). This lets the simulator's existing transfer-time
formula (\texttt{output\_size} $\div$ link bandwidth)
automatically account for the \textbf{KV-migration cost} if
Decode is scheduled to a different server than Prefill---no
additional code paths are required. Additionally, each phase
task carries a \texttt{kv\_cache\_GB} field that contributes
to GPU VRAM occupancy during execution, realistically modeling
memory pressure from active sequences.

\textbf{M3: Continuous Batching.} When a task is admitted to a
server, we count the number of currently running same-model
same-kind tasks (\texttt{batch\_size\_at\_admit}) and compute
its effective execution time as $T_{\text{solo}} \cdot
(1 + (\text{batch\_size} - 1) \cdot \text{overhead})$, with
phase-specific overhead constants (5\% per request for decode,
30\% for prefill, matching vLLM measurements). The server
enforces \texttt{max\_batch\_size} as admission control: a new
task arriving when the same-model same-kind batch is already
at capacity is rejected (\texttt{can\_allocate = False}) until
a slot frees. To distinguish batching's role from na\"ive
concurrent execution, our \texttt{--ablation no\_batching}
mode disables this mechanism \emph{and} enforces serial GPU
usage for inference tasks (one inference task per server at a
time), modeling the behavior of an inference engine without
iteration-level batching support.

\textbf{M4: Memory-Bandwidth Floor and Realistic Workloads.}
Real LLM decode is memory-bandwidth-bound: even with infinite
compute capacity, each output token takes a floor of $\sim$20~ms
on A100-class GPUs due to HBM bandwidth limits. We model this
by computing effective execution time as
$\max(\text{workload} \div \text{compute\_capacity},
\text{floor\_per\_token} \cdot \text{tokens})$, with per-model
floor constants (LLaMA-7B: 20~ms/token decode, 1~ms/token
prefill; scaling proportionally to model size up to LLaMA-70B's
80~ms/token decode). Without this floor, our simulator would
produce unrealistic sub-millisecond TPOT values that fail to
reflect the dominant production bottleneck. M4 also introduces
Poisson arrival modeling (adding an \texttt{arrival\_time}
field to each task that gates the
\texttt{WAITING}~$\to$~\texttt{READY} transition) and
log-normal prompt/output length distributions calibrated to
Azure LLM Inference Trace 2023 statistics
($\text{prompt } \mu_{\log}=5.5,\ \sigma_{\log}=1.0$;
$\text{output } \mu_{\log}=4.5,\ \sigma_{\log}=1.2$).

\textbf{Energy Model.} To support multi-objective evaluation,
the simulator tracks per-server energy consumption using a
linear-in-utilization power model:
$P(t) = \text{idle\_W} +
(\text{max\_W} - \text{idle\_W}) \cdot
\text{compute\_util}(t)$, with per-tier coefficients calibrated
to NVIDIA TDPs (cloud A100: 50--400~W; edge A100: 30--250~W;
T4: 20--70~W; Jetson AGX Orin: 10--30~W). Total energy is
accumulated as $E \mathrel{+}= P(t) \cdot \Delta t$ at each
simulation step; energy per token is computed as system-wide
total energy divided by completed output tokens. The
non-monotonic relation between compute capacity and power
efficiency (T4 has the best efficiency ratio despite mid-tier
compute) is crucial for our multi-objective evaluation---a
scheduler favoring ``always-cloud'' will have poor energy
efficiency even with low latency.

\textbf{Output Interface.} The simulator exposes a standard
\texttt{scheduler.schedule()} interface called once per
simulation step. Schedulers receive read-only access to the
full \texttt{Simulation} object (tasks, servers, network) and
must populate \texttt{task.assigned\_server} and call
\texttt{server.add\_task(task, priority)} for each
\texttt{READY} task they choose to dispatch. This minimal
interface allows any scheduler---from a 50-line round-robin
to our PPO agent---to be swapped in without simulator
modification, supporting fair controlled comparisons.

\textbf{End-to-end architecture.} Fig.~\ref{fig:simulator-arch}
summarizes the full simulator data flow: trace sources
(\texttt{uniform}/\texttt{lognormal} presets, \texttt{ModelCatalog})
feed a workload generator that emits prefill/decode DAGs;
the simulator's \texttt{step()} loop transitions tasks
\texttt{WAITING}~$\to$~\texttt{READY} as dependencies and
arrival times allow, invokes the scheduler, and dispatches
admitted tasks to heterogeneous servers (which model the four
AIGC physics M1--M4 plus the energy meter); per-checkpoint
common and AIGC metrics are then persisted alongside a
reproducibility manifest. The same architecture supports
every baseline and ablation in \S\ref{sec:eval}.

\subsection{AIGC-Aware Reinforcement Learning Scheduler}
\label{sec:rl}

Our scheduler casts request dispatch as a Markov Decision
Process where the agent makes a per-task discrete server
choice. Below we describe the state, action, and reward
designs that expose AIGC physics to the policy, followed by
the PPO training procedure.

\textbf{State Space.} For each \texttt{READY} task awaiting
dispatch, the encoder produces a flat feature vector of
dimension $(10 + |\mathcal{M}|) + 2 + 10 \cdot N$, where
$|\mathcal{M}|$ is the number of models in the catalog and $N$
is the number of servers. The vector concatenates three
blocks:

\begin{itemize}
\item \emph{Task block} ($10 + |\mathcal{M}|$ dims): compute
  demand, workload, input size, output size, dependency count,
  successor count, task-kind one-hot (Generic/Prefill/Decode),
  normalized KV-cache size, and a model-ID one-hot. The model
  one-hot is the most informative AIGC-specific feature,
  encoding \emph{which} model the task needs.

\item \emph{Global block} (2 dims): \texttt{READY}-queue size
  and completed-task ratio, providing horizon context.

\item \emph{Per-server block} (10 dims $\times N$ servers):
  five general-purpose features (compute utilization, memory
  utilization, bandwidth, queue length, transfer time estimate
  to this server) followed by five \textbf{AIGC-specific
  features}: (1) whether the current task's model is loaded
  on this server, (2) batch occupancy normalized to
  \texttt{max\_batch\_size}, (3) free VRAM fraction,
  (4) normalized cold-load cost for this task's model on this
  server, and (5) \emph{sibling-server indicator}---for Decode
  tasks, whether this server is where the corresponding
  Prefill ran.
\end{itemize}

The sibling-server indicator is the most novel feature: it
gives the policy direct read-access to KV-cache locality
without requiring it to reason about request IDs. For canonical
edge${=}5$, $|\mathcal{M}|{=}4$ models, the total state
dimension is 96.

\textbf{Action Space.} Discrete choice over the $N$ servers,
with \textbf{action masking} to zero out the probability of any
infeasible server (where \texttt{can\_allocate(task) = False}).
Masking prevents the agent from wasting samples on illegal
actions and stabilizes training. The mask is re-computed at
every dispatch decision.

\textbf{Reward Function.} A weighted sum of seven components,
balancing throughput, fairness, AIGC awareness, and energy:
\begin{equation}
  \label{eq:reward}
  \begin{aligned}
    r =\;& 0.25 \cdot r_{\text{time}}
        + 0.10 \cdot r_{\text{balance}}
        + 0.10 \cdot r_{\text{match}} \\
       &+ 0.15 \cdot r_{\text{warm}}
        + 0.20 \cdot r_{\text{batch}}
        + 0.10 \cdot r_{\text{affinity}} \\
       &+ 0.10 \cdot r_{\text{cloud}}.
  \end{aligned}
\end{equation}
The first three components are standard load-balancing
signals: $r_{\text{time}}$ favors low total execution time,
$r_{\text{balance}}$ discourages overloading any single
server, $r_{\text{match}}$ prefers servers whose compute
roughly matches task demand. The next three are AIGC-specific:

\begin{itemize}
\item $r_{\text{warm}}$ ($+1$ if model already loaded on chosen
  server; $-\text{cold\_load\_sec}/20$ otherwise, clipped to
  $[-1, 0]$) directly rewards avoiding cold loads.

\item $r_{\text{batch}}$ (proportional to existing same-model
  batch size on the chosen server, weighted higher for Decode
  whose batching is more efficient) rewards joining existing
  batches to capture throughput multipliers.

\item $r_{\text{affinity}}$ ($+1$ if chosen server hosts the
  sibling Prefill; $-0.5$ if not and the KV cache to migrate is
  non-trivial) rewards keeping KV cache local.
\end{itemize}

Finally, $r_{\text{cloud}}$ ($-\text{cloud\_util}$, applied
only when an edge alternative could serve the task) is a
diagnostic-driven addition that prevents the policy from
collapsing to an ``always-cloud'' attractor we observed in
early experiments. Without this term, PPO learned a degenerate
policy of routing nearly all traffic to the cloud server,
ignoring AIGC signals entirely (analyzed in our diagnostic
appendix). With it, the policy properly explores edge
alternatives and learns AIGC-aware placement.

\textbf{Policy Architecture.} A three-layer MLP encoder
($\text{state\_dim} \to 256 \to 128 \to 64$ with LayerNorm and
ReLU) produces a shared feature representation, which an Actor
head (Linear $\to$ softmax-with-mask) maps to action
probabilities and a Critic head (Linear $\to$ ReLU $\to$
Linear) maps to the value estimate. We deliberately use a
feedforward architecture rather than a graph or recurrent
encoder; our ablation (\S\ref{sec:eval}) and comparison against
the GNN variant show this choice does not sacrifice
performance under practical training budgets.

\textbf{Training Procedure.} We train with Proximal Policy
Optimization (PPO)~\cite{schulman2017ppo} using clipped
surrogate objective ($\epsilon = 0.2$), Generalized Advantage
Estimation~\cite{schulman2015gae} ($\gamma = 0.95$,
$\lambda = 0.90$), and entropy regularization (coefficient
$0.05$, reduced to $0.01$ after pretraining). The trajectory
buffer accumulates 32 transitions before each PPO update; each
update performs 4 epochs over the buffered data. Action
masking is applied during both action sampling and policy
log-probability computation.

\textbf{Pretraining.} A critical practical detail: we precede
online training with 20 episodes of pretrain on the target
workload. Pretrain is essential: without it, the policy
converges to a degenerate ``always-cloud'' attractor
(\S\ref{sec:eval} shows a $-38\%$ \slo{} drop from disabling
pretrain). With 20 pretrain episodes ($\approx$10--15 seconds
wall-clock), the policy escapes this attractor and learns
meaningful per-task placement decisions. We initialize pretrain
with 30 \emph{warmup} steps of uniform random action sampling
to encourage diverse experience collection before
policy-gradient updates begin.

\textbf{Ablation Interface.} Each component above is gated by
a constructor flag (\texttt{enable\_warm\_reward},
\texttt{enable\_aigc\_state}, etc.), enabling the systematic
12-component ablation analyzed in \S\ref{sec:eval}. Disabling
all AIGC components reduces our scheduler to a generic PPO
baseline operating only on base features---directly testing
whether AIGC awareness provides measurable value beyond
standard RL machinery.
